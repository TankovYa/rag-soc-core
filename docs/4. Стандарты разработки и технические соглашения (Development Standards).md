### 1.5 Naming conventions

| Объект | Правило | Пример |
|--------|---------|--------|
| Репозиторий | `rag-soc-<domain>` | `rag-soc-core`, `rag-soc-gateway` |
| Сервис (Python) | `rag-soc-<service>` | `rag-soc-converter`, `rag-soc-orchestrator` |
| Ветка | `<type>/<issue>-<short-name>` | `feature/RAG-42-add-pdf-converter` |
| Типы веток | `feature/`, `bugfix/`, `hotfix/`, `release/` | — |
| Коммиты | Conventional Commits | `feat(converter): add PDF support` |
| Теги | `v<semver>` | `v1.2.3` |
| Docker образ | `harbor.domain/rag-soc/<service>:<version>-<commit-hash>` | `harbor.domain/rag-soc/converter:1.2.3-a1b2c3d` |
| Helm чарт | `rag-soc-<service>` | `rag-soc-converter` |

---

### 1.6 Conventional Commits

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Типы:**
- `feat`: новая функциональность
- `fix`: исправление бага
- `docs`: документация
- `style`: форматирование, линтеры
- `refactor`: рефакторинг
- `test`: добавление тестов
- `chore`: CI/CD, конфигурация
- `perf`: оптимизация производительности

## 4. Логирование и трассировка

### 4.1 Уровни логирования (строго)

| Уровень | Использование | PROD разрешено |
|---------|---------------|----------------|
| `ERROR` | Критические ошибки, требующие вмешательства | ✅ Да |
| `WARN` | Проблемы без нарушения workflow (fallback, retry) | ✅ Да |
| `INFO` | Бизнес-события (индексация, запрос, ответ) | ✅ Да |
| `DEBUG` | Подробные данные (промпты, эмбеддинги) | ❌ Нет (только TEST) |

---

### 4.2 JSON логирование (обязательно)

Все сервисы обязаны выводить логи в JSON формате.

**Обязательные поля:**

```json
{
  "timestamp": "2026-03-25T10:30:00.123Z",
  "level": "INFO",
  "service": "rag-soc-orchestrator",
  "trace_id": "a1b2c3d4e5f6",
  "span_id": "g7h8i9j0k1l2",
  "user_id": "user-123",
  "message": "RAG request processed",
  "duration_ms": 1234
}
```

---

### 4.3 Python пример

```python
import json
import logging
import sys

class JSONFormatter(logging.Formatter):
    def format(self, record):
        log_entry = {
            "timestamp": self.formatTime(record),
            "level": record.levelname,
            "service": "rag-soc-converter",
            "message": record.getMessage(),
            "trace_id": getattr(record, "trace_id", None),
        }
        return json.dumps(log_entry)

handler = logging.StreamHandler(sys.stdout)
handler.setFormatter(JSONFormatter())
logging.basicConfig(handlers=[handler], level=logging.INFO)
```

### 4.5 Healthcheck и Readiness (обязательно)

Каждый сервис, предоставляющий API, обязан реализовать два endpoint:

| Endpoint  | Назначение                               | Использование              |
| --------- | ---------------------------------------- | -------------------------- |
| `/health` | Проверка, что сервис жив (liveness)      | Kubernetes liveness probe  |
| `/ready`  | Проверка готовности к обработке запросов | Kubernetes readiness probe |

---

#### 4.5.1 `/health` (Liveness Probe)

**Назначение:**

* Проверяет, что процесс жив
* НЕ проверяет внешние зависимости

**Требования:**

* всегда быстрый (< 50 ms)
* не делает сетевых вызовов
* не обращается к БД / Kafka / Redis

**Пример ответа:**

```json
{
  "status": "ok"
}
```

---

#### 4.5.2 `/ready` (Readiness Probe)

**Назначение:**

* Проверяет, может ли сервис обрабатывать запросы

**Обязательно проверяет:**

* DB (если используется)
* Redis (если используется)
* Kafka (если используется)
* внешние API (опционально, с timeout)

---

#### Минимальный ответ:

```json
{
  "status": "ready"
}
```

---

#### При проблемах:

```json
{
  "status": "not_ready",
  "reason": "redis_unavailable"
}
```

---

#### 4.5.3 Пример (Python / FastAPI)

```python
from fastapi import FastAPI
import asyncio

app = FastAPI()

@app.get("/health")
def health():
    return {"status": "ok"}

@app.get("/ready")
async def ready():
    try:
        # пример: проверка Redis
        await check_redis()
        
        # пример: проверка DB
        await check_db()

        return {"status": "ready"}
    except Exception as e:
        return {
            "status": "not_ready",
            "reason": str(e)
        }
```

---

#### 4.5.4 Kubernetes probes (обязательно)

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

---

#### 4.5.5 Правила (критично)

❌ Запрещено:

* использовать `/health` для проверки БД
* делать тяжёлые операции в `/ready`
* возвращать 200 при неготовности

✅ Обязательно:

* `/ready` возвращает **HTTP 503**, если не готов
* таймаут проверки зависимостей ≤ 1 сек
* логировать причины `not_ready`

---

#### 4.5.6 Минимальные проверки по типу сервиса

| Сервис       | Что проверять в `/ready`   |
| ------------ | -------------------------- |
| API Gateway  | downstream сервисы (ping)  |
| Search       | Elasticsearch + Vector DB  |
| Generator    | LLM доступность            |
| Orchestrator | Redis + Search + Generator |
| Indexer      | Kafka + Vector DB          |

## 7. CI/CD Pipeline

### 7.1 Стадии пайплайна (Jenkins / GitHub Actions)

```
stages:
  - lint
  - unit-test
  - integration-test
  - sast
  - build
  - sca
  - deploy-staging
  - dast
  - e2e-test
  - quality-test
  - deploy-prod
```

---

### 7.2 Пример Jenkinsfile (для rag-soc-core)

```groovy
pipeline {
    agent any
    
    stages {
        stage('Lint') {
            steps {
                sh 'ruff check services/ common/'
                sh 'mypy services/ common/ --strict'
            }
        }
        
        stage('Unit Test') {
            steps {
                sh 'pytest services/*/tests/unit/ --cov=. --cov-fail-under=80'
            }
        }
        
        stage('Integration Test') {
            steps {
                sh 'pytest services/*/tests/integration/ --docker-compose=docker-compose.test.yml'
            }
        }
        
        stage('SAST') {
            steps {
                sh 'semgrep --config auto .'
            }
        }
        
        stage('Build') {
            steps {
                script {
                    docker.build("harbor.domain/rag-soc/${SERVICE}:${env.GIT_COMMIT}")
                    docker.push("harbor.domain/rag-soc/${SERVICE}:${env.GIT_COMMIT}")
                }
            }
        }
        
        stage('SCA') {
            steps {
                sh 'pip-audit --requirement requirements.txt --severity high'
            }
        }
        
        stage('Deploy Staging') {
            steps {
                withKubeConfig([credentialsId: 'kubeconfig-staging']) {
                    sh """
                        helm upgrade --install rag-soc-${SERVICE} \
                            ./deployments/rag-soc-core \
                            --namespace staging \
                            --set image.tag=${env.GIT_COMMIT} \
                            --values deployments/rag-soc-core/values/staging.yaml
                    """
                }
            }
        }
        
        stage('DAST') {
            steps {
                sh 'zap-api-scan.py -t https://staging.rag-soc.domain/api/v1/openapi.json'
            }
        }
        
        stage('E2E Test') {
            steps {
                sh 'pytest tests/e2e/ --env=staging'
            }
        }
        
        stage('Quality Test') {
            steps {
                sh 'pytest tests/quality/ --ragas-metrics'
            }
        }
        
        stage('Deploy Prod') {
            when { branch 'main' }
            input { message 'Deploy to production?' }
            steps {
                withKubeConfig([credentialsId: 'kubeconfig-prod']) {
                    sh """
                        helm upgrade --install rag-soc-${SERVICE} \
                            ./deployments/rag-soc-core \
                            --namespace prod \
                            --set image.tag=${env.GIT_COMMIT} \
                            --values deployments/rag-soc-core/values/prod.yaml
                    """
                }
            }
        }
    }
}
```
